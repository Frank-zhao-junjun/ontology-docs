# MVP 验证阶段验收清单

版本：v1.0  
日期：2026-03-28  
用途：一页版评审、演示、阶段退出判断

## 1. 验证目标

本阶段只验证以下三件事：

1. 元模型发布后，运行时是否稳定生效。
2. AI 编排是否能稳定支撑查询、导航、受控写。
3. 前后端联调、失败降级、日志与测试是否足以支撑“可行”结论。

## 2. 当前范围

### 2.1 必须验证的闭环

| 闭环 | 验证点 | 当前状态 |
| --- | --- | --- |
| 元模型发布闭环 | 定义、校验、发布、回滚、快照切换 | 已具备基础能力 |
| AI 编排闭环 | 上下文、意图、策略、路由、响应构建 | 已具备基础能力 |
| 受控写闭环 | 规则校验、状态流转、事件触发 | 已具备基础能力 |
| 前端最小联调闭环 | chat 接口联通、结构化界面联动 | 已有骨架，需补齐 |
| 测试与可观测闭环 | pytest、关键错误可定位、关键日志可追踪 | 已有基础，需补齐 |

### 2.2 当前不验收的内容

1. 多租户、ABAC、复杂组织授权。
2. 跨系统事件总线、外部订阅投递。
3. 插件生态、模板市场、开放平台。
4. 多区域高可用、容灾演练、企业级运维治理。
5. AI 成本治理、企业合规体系、复杂 BPM 流程引擎。

## 4. 本阶段必须补齐项

| ID | 验收项 | 通过标准 |
| --- | --- | --- |
| MVP-004 | 统一失败响应结构 | 查询失败、规则失败、状态失败、事件失败返回统一结构 |

| MVP-006 | 端到端验证脚本 | 可一键验证模型发布、查询、受控写、降级、事件失败路径 |

## 5. 建议补齐项

| ID | 项目 | 通过标准 |
| --- | --- | --- |
| MVP-003 | 图表/分析结果最小展示面板 | 至少有一种稳定分析展示形式 |
| MVP-101 | 自愈修正链结构化记录 | 能看到失败原因、修正动作、最终结果 |
| MVP-102 | 性能采样脚本 | 能输出响应耗时统计，不作为 KPI 门槛 |
| MVP-103 | 日烟测脚本 | 可自动执行核心链路并输出结果 |
| MVP-104 | 审计日志字段补齐 | trace_id、session_id、user_id、error_code、event_id 可串联 |

## 6. 阶段通过标准

只有当以下条件全部满足，才能判断验证阶段通过：

1. 模型发布、查询、受控写、降级、事件派发五条主链路均可重复执行。
2. 前端能展示成功态、失败态、降级态。
3. 关键错误码可读，关键日志可追踪。
4. 至少存在一套端到端脚本稳定复现主链路。
5. pytest 与验证脚本在当前环境稳定通过。

## 7. 建议的单一冲刺范围

若只开一个 Validation Sprint，建议只做以下 5 项：

1. MVP-004 统一失败响应结构。
2. MVP-001 前端错误码映射。
3. MVP-002 最小合同域 CRUD 联调页面。
4. MVP-005 事件失败可观测补齐。
5. MVP-006 端到端验证脚本。
