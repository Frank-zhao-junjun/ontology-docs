# Ontology 本体建模简化方案设计文档

> 创建日期：2026-06-18
> 状态：待审阅

## 1. 设计目标

将当前 Ontology 本体建模工具从复杂的多维度、多层级能力，简化为一套清晰的“业务层级 + 8 维元模型”结构，同时引入模块级版本与确认机制，确保模型一致性可校验、可查询。

## 2. 核心设计决策

### 2.1 业务层级：严格父子树

采用线性链式结构，路径唯一：

```
A 业务价值域
└── B 业务能力
    └── C 业务场景
        └── D EPC 业务流程
```

- A/B/C/D 均为严格父子树。
- 一个 A 下包含多个 B；一个 B 下包含多个 C；一个 C 下包含多个 D。
- 任意 EPC 的完整路径可表示为：`A/B/C/EPC-01`。

### 2.2 元模型：8 个维度

| 维度 | 英文名 | 包含内容 |
|---|---|---|
| E1 数据模型 | Data Model | 实体、属性、关系、主数据 |
| E2 行为模型 | Behavior Model | 状态机、动作、约束（原实体生命周期并入此处） |
| E3 事件模型 | Event Model | 事件定义、订阅 |
| E4 规则模型 | Rule Model | 校验规则、业务规则 |
| E5 岗位角色模型 | Organization & Role Model | 部门、岗位、职责、治理角色 |
| E6 指标模型 | Metrics Model | 业务指标、计算口径 |
| E7 约束模型 | Constraint Model | 跨字段/跨实体/事务边界约束 |
| E8 接口模型 | Interface Model | 数据源、API、集成接口 |

### 2.3 8 维要素：全局共享 + 引用挂接

- 8 维要素在全局库中统一定义，不按 A/B/C 拆分。
- 每个要素记录业务来源路径：`valueDomainId` / `capabilityId` / `scenarioId`。
- EPC 通过引用挂接要素，要素侧维护反向索引用于校验。
- **要素都有稳定 ID 与可修改名称**：内部以 ID 为索引，用户界面显示名称。名称修改不影响引用关系。

### 2.4 EPC 编辑器与要素创建

- 在 EPC 编辑器中选择 8 维要素；若未找到，可直接新建。
- 新建的要素作为草稿进入对应 8 维模块库。
- 保存 EPC 时自动重建 8 维要素的反向索引。
- EPC 草稿可引用草稿态要素，但 **EPC 确认前必须先完成所引用要素的确认**。

## 3. 模块级版本与确认

### 3.1 版本规则

- A/B/C/D/E 每个模块独立拥有版本历史。
- 草稿无版本号；确认后生成 `v1`、`v2`、`v3`…
- 确认后再编辑，自动产生新版本，旧版本自动归档。
- 只有 **已确认版本** 中的要素才可被其他模块引用。

### 3.2 引用策略

- 默认引用 **最新已确认版本**。
- 支持 **锁定到指定版本**。
- 已有引用可选择“保持锁定版本”或“跟随最新已确认版本”。

### 3.3 版本数据模型

```ts
interface ModuleVersion<T> {
  id: string;           // 版本记录 ID
  elementId: string;    // 要素稳定 ID（跨版本不变）
  version: number;      // v1, v2, v3...
  status: 'draft' | 'confirmed' | 'archived';
  data: T;              // 模块具体内容
  createdAt: string;
  confirmedAt?: string;
  source?: 'manual' | 'excel_import' | 'ai_generate';
}
```

## 4. 校验与警示

### 4.1 校验时机

- 仅在模块 **确认时** 触发校验。
- 仅校验 **已确认版本**。

### 4.2 双向校验规则

| 方向 | 规则 | 结果 |
|---|---|---|
| D → E | EPC 引用的要素必须存在于 8 维模块库中 | warning |
| E → D | 每个已确认的 8 维要素必须至少被一个已确认 EPC 引用 | warning |

### 4.3 警示管理

- 提供可查询的 `WarningRegistry`。
- 支持按模块、按要素、按 EPC、按版本筛选。
- 确认操作被 warning 阻断，用户可选择忽略非致命 warning 或修复后重新确认。

## 5. Excel 导入导出

### 5.1 导出

- 每个模块（A/B/C/D/E1-E8）均可独立导出 Excel。
- A/B/C 各一个 Sheet；E 的每个维度一个 Sheet；D 一个 Sheet 并包含引用列。

### 5.2 导入

- 导入仅生成新的 **草稿版本**。
- 不直接覆盖或自动确认任何已存在数据。
- 导入后需经过人工审阅并确认，才能成为可引用版本。

### 5.3 Sheet 结构示例

| Sheet | 关键字段 |
|---|---|
| A-业务价值域 | `id`, `code`, `name`, `nameEn`, `parentId`, `semanticTags` |
| B-业务能力 | `id`, `code`, `name`, `nameEn`, `parentAId`, `semanticTags` |
| C-业务场景 | `id`, `code`, `name`, `nameEn`, `parentBId`, `semanticTags` |
| D-EPC流程 | `id`, `code`, `name`, `parentCId`, `usedEntityIds`, `usedRuleIds`… |
| E1-数据模型 | `id`, `code`, `name`, `entityNameEn`, `attributes`, `relations` |
| E2-行为模型 | `id`, `code`, `name`, `states`, `transitions`, `actions` |
| … | … |

## 6. AI 辅助建模

- A/B/C/D/E 任意模块的草稿版均可由 AI 辅助建模。
- AI 基于模块上下文（如 `A/B/C` 路径、已有 8 维要素）自动填充字段值。
- 人类可对 AI 生成的草稿进行编辑、修改、确认。
- 确认后再次编辑，自动产生新版本。

## 7. 旧能力处理

| 旧能力 | 处理方式 |
|---|---|
| Agent 语义层 | 删除，语义/别名能力下沉为 A/B/C 节点和 8 维要素的字段属性 |
| 实体生命周期 | 删除，生命周期状态机并入 E2 行为模型 |
| 流程模型（Orchestration） | 由 D-EPC 流程替代 |
| 主数据 | 并入 E1 数据模型 |
| 治理模型 | 并入 E5 岗位角色模型 |
| 数据源 | 并入 E8 接口模型 |
| 事务边界 | 并入 E7 约束模型 |

## 8. 引用方向

- **EPC 侧为主数据源**：EPC 记录其引用的 8 维要素 ID 列表。
- **要素侧维护反向索引（缓存）**：用于快速查询“某要素被哪些 EPC 引用”。
- EPC 保存时自动重建反向索引，保证双向查询一致性。

## 9. 待确认事项
