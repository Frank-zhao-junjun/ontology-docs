## 交付工作流（US → Unit）

本简化重构 **不按 Phase 大块交付**，而是：**Phase 仅作路线图；执行粒度为 User Story（US）与 Unit**。

### 原则

1. **流水线**（每个 Unit **必须六步严格按序走完**；任一步未绿灯不得标 Unit 完成）：
   ```
   US（用户确认）→ ① Unit Spec → ② PRD → ③ Testing case → ④ Coding → ⑤ Unit test → ⑥ E2E
   ```
   - **TDD 强制**：③ Testing case **必须先于** ④ Coding（可先 failing tests）
   - **E2E**：纯 lib/文档 Unit 须在 Spec 标明 `E2E: N/A` 并勾选
   - 门禁细则：[`UNIT_VALIDATION_CHECKLIST.md`](./ontology-simplification/UNIT_VALIDATION_CHECKLIST.md)
2. **拆分**：每个 US 拆成 **若干个 Unit**；**Unit 越小越好**（单一职责、少量文件、可独立验收、尽量一个 PR）。
3. **确认边界**：
   - **仅 US 需要用户确认**（范围、验收标准、优先级）→ 确认后才拆 Unit 并自动推进；
   - **Unit 内其余步骤自动执行**，无需逐步请示；
   - **Unit 成功 = 六步全部勾选 + 证据**（见下表）。

### Unit 绿灯门禁（`repo-main`）

| 步骤 | 产出 | 通过标准 |
|------|------|----------|
| Unit Spec | [`docs/ontology-simplification/units/{US-id}-U{n}-spec.md`](./ontology-simplification/units/_UNIT_SPEC_TEMPLATE.md) | 范围、接口、非目标、依赖写清 |
| PRD | Spec 内验收条款或链接 PRD 片段 | 与主计划决策无冲突 |
| Testing case | `repo-main/tests/...` 用例骨架或清单 | 用例可执行、覆盖 Spec 验收项 |
| Coding | `repo-main/src/...` | `pnpm lint` + `pnpm ts-check` 通过 |
| Unit test | 对应 `*.spec.ts` / `*.spec.tsx` | `pnpm test:unit` 相关用例 100% pass |
| E2E testing | 必要时 `tests/e2e/...` | `pnpm test:e2e:smoke` 或本 Unit scoped e2e 通过 |

整库发布前仍跑 `pnpm run ci:check`；单 Unit 至少保证 **lint + ts-check + 本 Unit 单元测试** 绿灯。

## 已确认的产品决策

| 议题 | 你的决定 |
|------|----------|
| B 与 C 关系 | **严格父子树**：线性链 `A → B → C`（一个 A 下多个 B，每个 B 下多个 C，C 只属于一个 B） |
| A/B 填写 | **不允许轻量分类**；A/B/C 均为必填完整结构 + 语义字段 |
| EPC ↔ 八维 | **双向引用校验仅警示、不阻断**；**只针对已确认版本** |
| 旧能力处置 | **Agent 语义层、实体生命周期等删除**，按新设计重做（语义沉入 A/B/C/D/E 字段） |
| 跨模块引用 | 默认引用 **目标模块最新已确认版本**；支持 **锁定指定版本**；草稿无版本号，确认后 `v1/v2…`，旧版自动归档，引用可保持或跟随最新 |
| **八维要素作用域** | **项目内全局共享**；不归属单一 C；通过 **引用挂接** 到 C 与 EPC；需独立 **要素归属/可见性** 管理；允许要素暂不挂 EPC（对已确认版本 **warning**） |
| **EPC 与 C 关系** | **一个 C 下可含多个 EPC**；树路径 `A/B/C/EPC-01`；EPC 为 C 的子节点 |
| **引用权威源** | **EPC 步骤 `elementRef` 为主数据**；要素库存定义；**`usageRefs` 派生反向索引** |
| **挂接 UX** | 日常在 **EPC 编辑器** 选八维要素；无匹配可 **内联新建**；**保存 EPC** 自动入库 + **重建索引** |
| **孤儿查询** | 要素库 `usageRefs.length === 0` → 未引用（W-EPC-02 + 筛选器） |
| **八要素标识** | 每项 **`id` 不可变** + **`name` 可改**；UI 见名称，引用/索引一律用 **ID** |
| **A/B/C/EPC 标识** | 同样遵循 **`id` 不可变 + `name` 可改**；引用/索引用 ID，UI 展示用名称 |
| **警示级别** | 所有 W-EPC 规则均为 **warning**，**不阻断**确认与导出；可在警示中心查询与忽略 |
| **Excel 导入父节点缺失** | 导入时若引用的父节点（A/B/C）不存在，**允许创建占位 draft**；确认前需补齐必填字段 |
| **交付工作流** | **US → Unit Spec → PRD → Testing case → Coding → Unit test → E2E**；US 拆多 Unit（越小越好）；**仅 US 需用户确认**，Unit 内自动执行，**全绿灯 = Unit 成功** |

## 核心数据模型（新增/修改 [`ontology.ts`](../repo-main/src/types/ontology.ts)）

### 业务链（均含语义块）

每种业务节点统一包含：

- `id`, `name`, `nameEn`, `description`
- `semantics`: `{ terms[], triggerPhrases[], synonyms[] }`（原 Agent 语义层能力下沉）
- `parentId`（B→A，C→B，**EPC→C**；A 无 parent）
- `moduleVersion`: 见下节

**EPC 节点**（`moduleKind: 'EPC'`）：

- 挂在某个 **C** 下；同一 C 可有 `EPC-01`、`EPC-02`…
- 树路径示例：`生产制造 / 计划能力 / MTS场景 / 订单下达流程`
- **`steps[].elementRef` 为引用权威源**（非独立挂接表）

### 八维要素库（E1–E8）— 项目级全局共享

八维要素 **不** 以 `scenarioId` 为所有权；统一存放在项目级 **要素库（Element Registry）**：

- **E1 数据**：`Entity` / `Attribute` / `Relation`
- **E2 行为**：`StateMachine` + `Action`
- **E3 事件**：`EventDefinition` + 订阅/路由
- **E4 规则**：`Rule`
- **E5 岗位角色**：Org/Position + governance.roles
- **E6 指标**：`BusinessMetric`
- **E7 约束**：`BehaviorConstraint` + 策略约束
- **E8 接口**：`DataSource` + 集成/Webhook

每个要素带：

- **`id`**：全局唯一、**创建后不可修改**；所有引用与索引的主键
- **`name`**：**可修改**；用户界面展示（选择器、列表、步骤标签）
- **`nameEn`**：可选、可改；Manifest / 平台导出
- `dimension`（E1–E8）、`visibility` / `ownership`
- **`usageRefs[]`（派生）**：由 `rebuildUsageIndex()` 生成；键为 `element.id`

### 八要素 ID / 名称规范

```
interface MetaElementBase {
  id: string;           // 不可变；EPC elementRef、usageIndex、编译器均引用此字段
  name: string;         // 可改；用户可见
  nameEn?: string;      // 可改；导出
  dimension: 'E1' | ... | 'E8';
}
```

- **展示层**：`resolveElementLabel(id) → name`（运行时查库，不缓存名称到 EPC 步骤）
- **存储层**：`EpcStep.elementRef.elementId = id` only
- **编辑约束**：要素库表单 `id` 只读；改名不影响已有 EPC 引用
- **内联新建**：自动生成 `id` + 用户输入 `name` → EPC save 时 upsert 入对应 E 维 draft

### 要素归属 / 可见性 + 反向索引

```
type ElementVisibility = 'project' | 'domain_scoped' | 'private_draft';

interface EpcStep {
  id: string;
  name: string;
  elementRef?: {
    dimension: 'E1' | 'E2' | ... | 'E8';
    elementId: string;
    versionPin: 'latest_confirmed' | { version: string };
    inlineNew?: boolean;        // 选择器内新建，EPC save 时 upsert 入八维 draft
    inlinePayload?: unknown;    // 新建时的最小字段快照
  };
}

interface MetaElement {
  id: string;                     // ★ 不可变
  name: string;                   // ★ 可改，UI 展示
  nameEn?: string;
  dimension: 'E1' | 'E2' | ... | 'E8';
  visibility: ElementVisibility;
  ownerModuleId?: string;
  confirmedVersion?: string;
  usageRefs?: ElementUsageRef[];  // 派生；按 id 聚合
}

interface ElementUsageRef {
  epcId: string;
  stepId: string;
  scenarioId: string;             // 冗余 C.id
  versionPin: 'latest_confirmed' | { version: string };
}
```

**Store API**：

- `saveEpcDraft(epc)` → upsert `inlineNew` 要素到对应 E1–E8 draft → `rebuildUsageIndex()`
- `getUnreferencedElements()` → `usageRefs.length === 0` 的已确认要素
- `getElementUsage(elementId)` → 只读展示引用来源（EPC/步骤）

- **要素库 UI**：按 E1–E8 浏览、独立 CRUD、确认版本；**「未引用」筛选**
- **C 工作区**：不维护独立挂接表；展示子 EPC **引用并集**（只读）
- **EPC 工作区（主入口）**：步骤编辑 + 八维选择器（可内联新建）

### 模块版本与确认状态

```
type ModuleKind = 'A'|'B'|'C'|'EPC'|'E1'|'E2'|...|'E8';
type ModuleStatus = 'draft' | 'confirmed' | 'archived';

interface ModuleVersionRecord {
  moduleKind: ModuleKind;
  moduleId: string;           // 业务对象 id
  status: ModuleStatus;
  version?: string;             // 草稿无；确认后 v1, v2...
  confirmedAt?: string;
  snapshot: unknown;            // 该模块完整 JSON 快照
  parentVersionRefs?: RefPin[]; // 引用的其他模块版本
}

interface CrossModuleRef {
  targetModuleKind: ModuleKind;
  targetElementId: string;
  pin: 'latest_confirmed' | { version: string };
}
```

**规则**：

- 编辑已确认内容 → 自动 fork 新 **draft**（不覆盖 vN）
- 确认 draft → 生成 `v(N+1)`，上一确认版 → `archived`
- **引用门禁**：仅 `confirmed` 快照内要素可被跨模块引用；draft 仅本模块内编辑
- 默认 `pin: latest_confirmed`；UI 提供「锁定到 v2」

## 校验与警示（第 3 条）

新增 [`business-epc-linter`](../repo-main/src/lib)（建议路径）：

| 规则 ID | 条件 | 级别 | 作用范围 |
|---------|------|------|----------|
| W-EPC-01 | EPC 步骤引用的 `elementId` 不在八维 **已确认** 要素集 | warning | EPC confirmed |
| W-EPC-02 | 八维 **已确认** 要素 `usageRefs` 为空（未被任何 EPC 步骤引用） | warning | E* confirmed |
| W-EPC-03 | 步骤引用目标要素仅有 draft、无 confirmed | warning | 跨模块 ref |
| W-EPC-04 | C 下无任何 EPC 子节点 | warning | C confirmed |
| W-EPC-05 | 步骤 `elementId` 在八维库中不存在 | warning | 任意 confirmed |

**要点**：

- 八维要素 **允许** 暂未被任何 EPC 引用（如先在库中建好）→ **W-EPC-02** + 要素库「未引用」筛选
- EPC 步骤 **不允许** 引用未确认要素 → **W-EPC-01**（仍仅 warning，不阻断确认/导出）
- 校验 **仅针对已确认版本** 的快照；draft 不跑 W-EPC-02/04
- **不阻断**确认与导出；在「警示中心」可查询（复用 manifest-export 列表 UX）

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 严格树限制真实业务复用 | 允许 **复制 C/EPC 到另一 B/C**；八维要素全局复用靠 attachment |
| 全局要素库导致孤儿要素 | W-EPC-02 + **`usageRefs` 未引用筛选**（索引由 EPC save 重建） |
| 索引与 EPC 不一致 | `rebuildUsageIndex` 为幂等全量扫描；导出/linter 前可强制重建 |
| 多 EPC 与 compiler 导出范围 | 导出时按 C 子树或全项目 **闭包收集** 被引用要素 |
| 校验逻辑复杂 | attachment 与要素版本分离；linter 只读 confirmed 快照 |
| 大 bang 迁移 | Phase 1 双写；旧 `businessScenarioId` → attachment 表 |
| 与 ontology-platform 契约 | compiler 映射表评审 + 制造域 golden 扩展 |

## 成功标准

- 用户路径：建 A → B → C → EPC → **在 EPC 步骤上挂接/新建要素** → 保存自动入库
- 要素库可筛 **未引用** 要素；引用关系来自派生 `usageRefs`
- 跨模块引用仅指向 confirmed；默认 latest，可 pin
- 对已确认版本：EPC↔八维双向 warn 可查询，不阻断导出
- Excel/AI 仅产生 draft
- Agent 语义层 / Entity Lifecycle 无独立入口；能力在新字段可表达
- `pnpm run ci:check` 通过，Manifest golden 不退化
- 每个已交付 Unit：Spec → E2E 流水线 **全绿灯**
