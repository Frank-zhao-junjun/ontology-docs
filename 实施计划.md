# 本体驱动 AI 原生应用框架实施计划

版本：v1.0  
日期：2026-03-28  
对应需求基线：最终需求文档 v2.2

## 1. 计划目标

本实施计划用于将需求基线转化为可执行交付，覆盖：

1. 里程碑与节奏（2-4 周粒度）。
2. 每阶段交付物。
3. 可验证的验收标准。
4. 优先级与风险控制。

## 2. 实施原则

1. 先闭环后扩展：优先打通“元模型定义 -> 生成 -> 发布 -> 运行”。
2. 先核心后体验：先实现受控执行链路，再优化交互和分析体验。
3. 先单域后多域：首个验证域为合同管理。
4. 需求优先级遵循 P0/P1/P2：
   1. P0：MVP 必须项，不完成不进入下一阶段。
   2. P1：提升完整性，建议在 MVP 内完成。
   3. P2：增强项，可后置。

## 3. 里程碑总览（建议 14 周）

| 里程碑 | 周期 | 核心目标 |
| --- | --- | --- |
| M1 基础工程与基线冻结 | 第 1-2 周 | 建立仓库骨架、开发规范、需求追踪矩阵、环境可启动 |
| M2 元模型与生成链路 | 第 3-5 周 | 五类元模型最小可用、生成与发布流程打通 |
| M3 运行时核心闭环 | 第 6-9 周 | AI 编排器、领域技能、规则/状态机、事件订阅闭环 |
| M4 混合交互与验收场景 | 第 10-12 周 | 三栏交互、动作协议联动、合同域端到端场景通过 |
| M5 稳定性与发布准备 | 第 13-14 周 | 缺陷收敛、演示验收、文档与发布包齐套 |

## 4. 分阶段实施明细

## M1 基础工程与基线冻结（第 1-2 周）

### 目标

1. 形成统一工程骨架与协作规范。
2. 锁定需求基线映射关系（需求条款 -> 实施任务 -> 验收用例）。
3. 完成本地一键启动和最小 CI。

### 关键任务

1. 建立目录结构与代码规范（后端、前端、生成物、元模型源）。
2. 建立需求追踪矩阵（RTM）。
3. 建立基础 CI（静态检查、单元测试占位、构建检查）。
4. 定义分支策略、发布策略、版本号规则。

### 交付物

1. 项目结构初始化提交。
2. 需求追踪矩阵（requirements-traceability.md）。
3. 开发规范文档（编码规范、提交流程、评审标准）。
4. 本地启动与 CI 说明文档。

### 验收标准

1. 新成员可按文档在 30 分钟内完成环境启动。
2. CI 能在主干分支自动执行并给出状态结果。
3. 需求追踪矩阵覆盖最终需求文档全部章节条款。

### 优先级

1. P0：需求追踪矩阵、环境启动、CI 骨架。
2. P1：模板化 PR 检查项。
3. P2：自动化脚手架生成优化。

## M2 元模型与生成链路（第 3-5 周）

### 目标

1. 完成五类元模型（数据、行为、规则、流程、事件）最小定义与版本化。
2. 完成“配置 -> 生成 -> 发布”受控流程。
3. 完成合同域基础实体生成。

### 关键任务

1. 定义五类元模型 Schema 与版本字段。
2. 实现元模型存储、发布校验与发布状态管理。
3. 实现生成器：数据结构、接口骨架、页面骨架、事件注册占位。
4. 建立发布流水：发布前校验、生成、发布后索引/注册。

### 交付物

1. 元模型 Schema 文档与示例。
2. 元模型管理 API 或管理入口。
3. 生成器最小可用版本。
4. 合同域首批实体（客户、员工、产品、合同、发票）生成产物。

### 验收标准

1. 能新增实体定义并完成发布，自动产生对应后端与前端骨架。
2. 能定义事件类型和订阅关系并通过发布校验。
3. 生成产物能追溯到元模型版本。

### 优先级

1. P0：数据模型、行为模型、规则模型、发布流程。
2. P1：流程模型与事件模型生成占位。
3. P2：生成模板扩展能力与代码美化。

## M3 运行时核心闭环（第 6-9 周）

### 目标

1. 打通 AI 编排器与工具执行闭环。
2. 打通领域技能与规则/状态机/事件联动。
3. 实现受控执行与失败可观测。

### 关键任务

1. AI 编排器：上下文管理、意图识别、工具路由、响应整合。
2. 工具体系：只读查询工具、技能调用工具、UI 动作工具、图表工具。
3. 规则引擎：五种规则类型的执行框架。
4. 行为模型执行器：简化状态机与流转校验。
5. 事件模型运行时：先进程内订阅触发、失败记录。
6. 自愈机制：最多 2 次重试，失败透明返回。

### 交付物

1. AI 编排器服务与工具适配层。
2. 规则引擎与状态机执行模块。
3. 领域技能基类和合同域核心技能。
4. 事件发布与订阅处理模块（进程内）。
5. 运行日志与错误可观测基础。

### 验收标准

1. 自然语言查询可稳定生成只读查询并返回结果。
2. 写操作均通过技能执行，直接写 SQL 被阻断。
3. 状态流转时规则校验生效，失败有明确反馈。
4. 典型业务操作后可触发并记录事件订阅处理结果。
5. 可修复错误在 2 次内完成自愈，超限时明确报错。

### 优先级

1. P0：编排器主链路、规则/状态机、技能受控执行。
2. P1：事件订阅异步化与更细日志。
3. P2：高级提示与智能纠错增强。

## M4 混合交互与验收场景（第 10-12 周）

### 目标

1. 完成三栏式混合交互界面。
2. 完成对话与 UI 动作协议联动。
3. 通过合同域端到端关键场景。

### 关键任务

1. 三栏布局与核心页面（列表、表单、详情、图表、欢迎页）。
2. 动作协议消费层（打开页面、刷新、聚焦、消息通知、渲染图表）。
3. 对话状态、聚焦实体、上下文面板联动。
4. 合同域验收场景脚本化（查询、录入、状态流转、事件演示）。

### 交付物

1. 混合交互前端最小可用版本。
2. 动作协议实现文档与示例。
3. 端到端验收脚本与录屏说明。

### 验收标准

1. 用户可通过自然语言触发页面打开与业务动作。
2. 对话反馈与界面状态一致。
3. 验收场景可一键执行并输出结果报告。

### 优先级

1. P0：三栏框架、核心页面、动作协议。
2. P1：图表体验优化。
3. P2：高级可视化和交互动效。

## M5 稳定性与发布准备（第 13-14 周）

### 目标

1. 收敛缺陷并完成上线前检查。
2. 形成可演示、可交付、可复现实验包。

### 关键任务

1. 缺陷分级与关单（阻断类、严重类优先）。
2. 关键链路回归测试与冒烟测试。
3. 运维与部署说明、备份恢复演练。
4. 演示脚本和验收报告整理。

### 交付物

1. 发布候选版本（RC）。
2. 测试报告（功能、回归、已知问题）。
3. 部署与运维手册。
4. 项目验收报告。

### 验收标准

1. P0/P1 需求项完成率达到约定阈值（建议 P0 100%，P1 >= 85%）。
2. 阻断类缺陷清零。
3. 端到端关键场景连续执行通过率满足约定阈值（建议 >= 95%）。

### 优先级

1. P0：缺陷清零、发布包、验收报告。
2. P1：文档齐套与运维演练。
3. P2：后续版本建议清单。

## 5. 角色与职责建议

| 角色 | 主要职责 |
| --- | --- |
| 产品负责人 | 需求基线维护、验收口径确认、优先级裁剪 |
| 技术负责人 | 架构决策、里程碑推进、技术风险收敛 |
| 后端负责人 | 元模型、生成链路、运行时与技能实现 |
| 前端负责人 | 混合交互、动作协议消费与页面落地 |
| AI 工程负责人 | 编排器策略、提示与工具调用质量 |
| QA 负责人 | 验收用例、回归体系、质量门禁 |

## 6. 风险与应对

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 生成链路与运行时模型不一致 | 高 | 强制版本追溯，发布前一致性校验 |
| AI 工具调用不稳定 | 高 | 工具参数约束、自愈重试、错误分级 |
| 事件订阅执行不可观测 | 中 | 统一日志结构、订阅执行结果打点 |
| 前后端动作协议频繁变更 | 中 | 先冻结协议 v1，变更走评审 |
| 场景过多导致 MVP 失焦 | 高 | 严格按 P0/P1 交付，P2 后置 |

## 7. 计划检查点（每周）

每周例会至少检查以下 6 项：

1. 里程碑是否按周目标推进。
2. P0 需求是否存在阻断项。
3. 缺陷趋势是否可控。
4. 生成链路与运行时一致性是否破坏。
5. 端到端验收脚本是否持续可执行。
6. 需不需要触发范围裁剪或资源调整。

## 8. 第一周立即行动清单

1. 冻结需求基线版本号并建立需求追踪矩阵。
2. 创建实施看板（按 M1-M5 与 P0/P1/P2 维度）。
3. 完成工程初始化和最小 CI。
4. 确认合同域首批验收场景清单。
5. 排定 M1 结束评审会议与准入标准。
