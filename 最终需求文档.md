## 1. 文档概述

### 1.1 文档目的

本文档定义“本体驱动的 AI 原生应用框架”的需求基线。该系统面向企业级业务场景，通过元模型统一描述业务对象、行为、规则、流程与领域事件，并由 AI 参与开发生成与运行时编排。

本文档聚焦需求、边界与验收标准。
具体技术栈与参考实现见**附录 B**，推荐目录结构示例见**附录 C**；附录内容均为推荐性质，**不构成**强制约束。

### 1.2 目标

系统应支持以下核心目标：

1. 通过统一元模型描述业务语义，降低业务系统构建成本。
2. 支持 AI 理解元模型并进行查询、辅助录入、分析和受控操作。
3. 支持基于模型生成基础应用骨架，减少重复开发。
4. 支持在运行期通过语义注入与工具调用，实现 AI 驱动的混合交互体验。
5. 保证关键业务动作具备可校验、可约束、可追踪的执行机制。

### 1.3 目标读者

1. 产品经理
2. 业务架构师
3. 技术架构师
4. 后端与前端工程师
5. AI 应用工程师
6. 领域专家

### 1.4 术语定义

| 术语 | 定义 |
| --- | --- |
| 元模型 | 对业务语义的某一类形式化描述；五类元模型为数据模型、行为模型、规则模型、流程模型、事件模型，共同构成完整本体 |
| 本体 | 由五类元模型共同表达的领域语义整体；亦可指某一业务对象（如合同）在其上挂载的数据、行为、规则、流程与事件等模型的集合 |
| 事件模型 | 描述领域事件类型、载荷结构及订阅关系的元模型；首个版本仅覆盖领域事件与订阅 |
| 领域技能 | 面向业务动作的后端能力单元 |
| AI 编排器 | 负责意图理解、上下文组织、工具选择和结果整合的运行时中枢 |
| 混合交互 | 自然语言交互与结构化界面协同工作的交互方式 |
| 生成期 | 基于元模型生成应用骨架、接口、页面和配置的阶段 |
| 运行期 | 基于上下文和工具调用执行查询、分析和业务操作的阶段 |
| 语义注入 | 将元模型和业务上下文组织为 AI 可理解输入的机制 |

## 2. 产品定位与范围

### 2.1 产品定位

本系统是一个面向企业业务场景的通用型应用框架，不是单一合同系统。合同管理场景仅作为首个验证域。

### 2.2 范围内能力

1. 基于元模型定义业务实体、属性、关系和约束。
2. 基于行为模型定义状态与状态流转。
3. 基于规则模型定义校验与约束规则。
4. 基于流程模型定义 AI 编排的上下文策略与受控动作。
5. 基于事件模型定义领域事件类型与订阅关系（见 4.6）。
6. 生成基础 CRUD 页面、接口和服务骨架。
7. 在运行期支持查询、分析、导航、辅助录入和受控操作。
8. 提供混合交互界面，支持对话与结构化 UI 联动。
9. 记录 AI 交互、工具调用和关键业务状态变化。

### 2.3 范围外能力

1. 不要求在首个版本支持复杂 BPMN 工作流引擎。
2. 不要求在首个版本支持移动端完整适配。
3. 不要求在首个版本支持高并发、多租户、大规模分布式部署。
4. 不要求在首个版本支持完全无发布的前后端即时动态扩展。

### 2.4 MVP 身份与权限边界

1. **认证**：首个版本要求**登录后访问**业务与 AI 能力；认证实现可采用框架内置账号体系，或对接企业统一身份（如 OIDC、SAML、LDAP 等），具体选型在实现阶段确定，不作为本文档强制技术约束。
2. **授权**：应支持基于**角色**或等价抽象（如权限组）控制对实体数据、领域技能及 AI 可调工具集合的访问；关键写操作与敏感查询须在授权范围内执行。
3. **范围外（首个版本不作为验收项）**：细粒度 ABAC、多租户数据隔离、与复杂组织模型绑定的动态授权策略；与 **2.3** 中「多租户」范围外表述一致，可在后续版本扩展。
4. **与正文其他条款的关系**：「权限机制」「权限校验」均指上述 MVP 范围内的身份与授权；若部署环境对传输加密、存储加密或合规等级有额外强制要求，在**部署与运维方案**中满足，不作为本文档单独成章的必选需求条款。

## 3. 总体需求

### 3.1 总体架构原则

系统应采用三层结构：

1. 元模型层：定义业务语义。
2. AI 引擎层：负责理解、推理、调用工具与执行约束。
3. 混合交互层：承载对话、结构化页面和上下文辅助。

### 3.2 生命周期要求

系统应覆盖两个阶段：

1. 生成期：基于元模型生成数据库结构、接口、服务骨架和前端页面骨架；当元模型中包含事件类型与订阅定义时，应生成或注册与之对应的基础骨架（如事件类型注册、订阅与领域技能的绑定占位、发布调用入口等），具体形态由实现约定。
2. 运行期：基于元模型与上下文进行语义注入、工具调用和受控执行。

## 4. 元模型层需求

### 4.1 通用要求

1. 五类元模型均应具备唯一标识、名称、版本、所属领域、定义内容、创建时间、更新时间。
2. 元模型应支持版本化管理。
3. 元模型变更应具备发布流程，不得直接绕过版本控制进入生产可用状态。
4. 元模型应可被 AI 编排器读取并转换为运行期语义上下文。

### 4.2 数据模型需求

#### 4.2.1 实体能力

1. 应支持定义实体、属性、关系、索引。
2. 应支持基础类型、引用类型、枚举类型、集合型字段。
3. 应支持一对一、一对多、多对多关系。
4. 应支持通过配置新增业务实体。

#### 4.2.2 动态扩展边界

1. 新增实体属于“配置、生成、发布”的受控流程。
2. 后端与数据层应支持根据元模型扩展实体结构。
3. 前端不要求在无发布的情况下即时支持新实体页面。
4. 新增实体后，应通过生成和发布流程补齐对应页面、接口和服务能力。

#### 4.2.3 持久化要求

1. 元数据应统一管理。
2. 业务实体数据应采用独立物理表存储，而不是统一 JSON/EAV 业务数据表作为主方案。
3. 多实体关系应支持明确的外键或等价关系约束机制。
4. 查询能力应面向结构化 SQL 与索引优化设计。

### 4.3 行为模型需求

#### 4.3.1 范围定义

1. 首个版本采用简化状态机模型。
2. 不支持并行状态、子状态、复杂会签等高级工作流能力。
3. 状态流转由显式业务动作触发。

#### 4.3.2 模型结构

行为模型应至少支持：

1. 状态机定义。
2. 初始状态与终态定义。
3. 状态流转定义。
4. 每个流转的触发方式。
5. 与流转绑定的校验和动作。

#### 4.3.3 语义约束

1. 状态机表达采用简化版语义，不引入 `pre_conditions`、`pre_actions`、`post_actions` 作为正式需求字段。
2. 执行引擎应围绕统一的流转定义与动作绑定实现，不得与模型结构脱节。
3. 状态变化应被记录并可追踪。

### 4.4 规则模型需求

#### 4.4.1 规则类型

系统应支持下列**五种规则类型**（指校验/约束的分类，与「五类元模型」不同）：

1. 字段级校验。
2. 跨字段校验。
3. 跨实体校验。
4. 聚合校验。
5. 时序规则。

#### 4.4.2 规则管理

1. 规则应支持存储、启停、版本化和发布。
2. 规则变更后应支持热加载或等价的低中断生效机制。
3. 规则引用必须使用统一且可解析的规则 ID。
4. 文档中的金额正数规则统一命名为 `amount_positive`。

### 4.5 流程模型需求

1. 流程模型用于定义 AI 编排策略，不要求采用 BPMN 图作为唯一表达形式。
2. 流程模型应定义入口、上下文收集步骤、校验步骤、决策点和完成动作。
3. 关键业务节点必须可由规则模型和权限机制约束。
4. AI 可参与流程决策，但不得绕过硬性规则和权限限制。
5. 首个版本的流程模型**面向 AI 编排语境**（上下文收集、校验、决策与受控动作）；**不**作为企业级人工审批链、会签或通用 BPM 工作流的完整表达载体，与 **2.3** 范围外能力一致。需人工参与的业务约束优先通过行为模型、规则、权限与界面协作实现，而非要求本流程模型承担 BPMN 级人工流。

### 4.6 事件模型需求

#### 4.6.1 范围定义

1. 事件模型用于描述**领域事件**及**订阅**关系，与数据、行为、规则、流程模型并列，共同构成完整本体。
2. 首个版本**仅**要求支持**领域事件**与**订阅**，不要求通用消息中间件、跨系统集成事件总线、复杂事件处理（CEP）等能力。
3. 不要求在首个版本将系统审计日志、AI 交互日志等运行期日志纳入事件模型管理；上述日志仍按 8.3 等条款单独保留。

#### 4.6.2 模型结构

事件模型应至少支持：

1. 领域事件类型定义（含事件名称及载荷结构，或与数据模型中结构的关联方式）。
2. 订阅关系定义：订阅方、所订阅的事件类型、处理语义（如关联的领域技能或内部处理入口的声明方式）。
3. 与行为模型或领域技能的衔接：业务动作或状态流转可声明触发领域事件发布。

#### 4.6.3 语义约束

1. 事件模型定义应可版本化，变更应遵循 4.1 中的发布流程，不得绕过版本控制进入生产可用状态。
2. 事件订阅的运行时行为以已发布的事件模型定义为准；首个版本不要求事件定义与订阅关系具备与规则模型同等的独立热加载能力。

#### 4.6.4 订阅方与投递范围（首个版本）

首个版本采用**先进程内**策略：领域事件的订阅声明、触发与处理均在**同一应用部署单元、同一进程空间**内完成（实现若使用进程内队列派发，仍视为进程内范畴）；**对外系统通知与跨部署集成**不作为本版本验收范围。

1. **订阅方**：在**同一应用部署单元、同一进程空间内**可被解析并执行的处理目标，且须在元模型或框架注册信息中**显式声明**，运行期不得依赖未在发布物中出现的动态外部地址。
2. **首选绑定**：订阅关系绑定到**领域技能**（技能入参承载事件类型标识与载荷，或采用双方约定的标准封装）；若采用等价的**命名处理入口**，须在发布物中可解析、可追溯，与「未声明即调用」相区别。
3. **范围外（首个版本不要求）**：向**外部 HTTP/Webhook**、**跨应用或跨部署单元**、**独立消息中间件上的第三方消费者**注册订阅或投递事件；该类集成归入后续版本或集成层，不由事件模型首个版本承担。
4. **多对多**：同一事件类型可对应多个订阅方；同一订阅方可订阅多种事件类型；**触发顺序**与**失败隔离**（单订阅失败是否影响其他订阅）由实现约定，但须在运维或日志层面可区分单次订阅执行结果。
5. **执行语义**：至少须支持订阅触发后对绑定处理逻辑的**一次调用**；同步调用或经进程内队列异步派发由实现选择。订阅处理失败时须**可记录、可观测**，不得静默丢弃；是否重试、是否影响发布领域事件的主事务，由实现与事务策略约定，需求层要求与业务写操作的风险边界在设计与评审中明确。

## 5. AI 引擎层需求

### 5.1 AI 编排器能力

AI 编排器应具备以下能力：

1. 上下文管理。
2. 意图识别。
3. 参数提取。
4. 元模型语义注入。
5. 工具选择与调用。
6. 错误反馈与有限重试。
7. 结果整合与对话响应。

### 5.2 上下文管理要求

1. 应维护会话历史。
2. 应维护当前聚焦实体或当前业务上下文。
3. 应在工具调用前构造与当前任务相关的最小必要上下文。
4. 应避免将无关上下文注入 AI，降低错误率和成本。

### 5.3 模型能力要求

AI 模型要求采用厂商无关的能力约束，正文不绑定单一供应商。

模型应满足：

1. 支持工具调用或等价函数调用能力。
2. 支持结构化输出。
3. 支持可配置模型切换。
4. 支持低温度、可控性较高的推理设置。
5. 支持超时、重试和失败降级机制。

### 5.4 工具体系要求

系统至少应支持以下工具类别：

1. 数据查询工具。
2. 领域技能调用工具。
3. UI 路由与界面动作工具。
4. 图表或可视化生成工具。
5. 直接响应工具。

### 5.5 安全与受控执行

1. AI 不得直接执行未受控写操作。
2. 查询工具默认仅允许只读查询。
3. 写操作必须通过受控技能执行。
4. 工具调用前后应记录输入、输出、错误和重试信息。

### 5.6 自愈机制要求

1. 系统应支持对工具执行错误进行有限次数重试。
2. 首个版本重试次数不应超过 2 次。
3. 重试仅用于参数、字段名、查询结构等可修复错误。
4. 权限不足、规则阻断、业务校验失败等问题不得被自愈机制绕过。

## 6. 领域技能需求

1. 领域技能应具备明确名称、参数结构、执行入口和返回结果结构。
2. 技能应独立承载业务写操作和复杂业务逻辑。
3. 技能执行前应进行参数校验。
4. 技能执行时应联动规则引擎与状态机。
5. 技能应支持权限校验。
6. 技能执行结果应具备成功、失败和错误明细表达。
7. **与事件模型衔接**：技能可被声明为领域事件订阅的处理入口，或在完成受控写与状态流转后按事件模型**发布**领域事件；订阅触发下的技能执行仍须满足本条第 3～5 款，事件类型、订阅绑定与进程内边界见 **4.6**。

## 7. 混合交互层需求

### 7.1 交互模式

系统应支持自然语言交互与结构化界面协同工作：

1. 用户可通过自然语言发起查询、分析、导航和操作意图。
2. 系统可根据意图打开列表、表单、详情、图表等界面。
3. 对话结果与界面状态应保持一致。

### 7.2 布局要求

首个版本应采用三栏式桌面布局：

1. 左栏承载对话历史与输入。
2. 中栏承载结构化内容区。
3. 右栏承载上下文辅助与关联信息。

### 7.3 页面能力

系统至少应支持以下页面类型：

1. 欢迎页。
2. 实体列表页。
3. 实体表单页。
4. 实体详情页。
5. 仪表盘页。
6. 图表展示页。

### 7.4 前后端动作协议

1. AI 返回结果应支持消息与动作分离表达。
2. 前端应支持打开页面、弹窗、导航、刷新、聚焦实体、展示消息、渲染图表等动作。
3. 动作协议应具备可扩展性。

### 7.5 可视化能力

系统应支持：

1. 统计类图表。
2. 流程图或关系图。
3. 表格视图。
4. 面向分析场景的结构化结果展示。

## 8. 非功能性需求

### 8.1 性能与 MVP 边界

1. **MVP 阶段不对性能数字设定验收性 KPI**：不将响应时间、首屏时间等作为必须通过或否决交付的量化门槛；亦不要求在 MVP 中约定统一测量环境与基准数据集。
2. 下表仅作为**体验与后续优化的参考方向**；若正式版本需要可量化的 SLA 或 KPI，应在 MVP 通过后的版本中另行定义测量方法与验收条件。

| 指标 | 参考方向（非 MVP 验收 KPI） |
| --- | --- |
| 页面首屏加载 | 小于 3 秒 |
| 简单 AI 查询响应 | 小于 5 秒 |
| 复杂 AI 分析响应 | 小于 15 秒 |
| 常规列表页加载 | 小于 1 秒 |
| 表单提交响应 | 小于 2 秒 |

### 8.2 可用性要求

1. AI 服务不可用时，系统应可降级为纯 UI 模式。
2. 系统应支持数据备份与导出迁移。
3. 用户应能感知 AI 当前执行状态。

### 8.3 可维护性要求

1. 元模型、规则、流程与事件模型定义应可版本化。
2. 生成产物应可追溯到对应元模型版本。
3. 系统应保留 AI 交互、工具调用和状态变化日志。

### 8.4 兼容性要求

1. 首个版本面向桌面浏览器。
2. 桌面端支持主流现代浏览器。
3. 移动端不作为首个版本正式支持范围。

## 9. 验收标准

### 9.1 元模型与生成

1. 管理员可定义新业务实体并完成发布流程。
2. 发布后系统可生成对应的数据结构、接口和基础页面骨架。
3. 管理员可定义领域事件类型及订阅关系，并在发布后对典型业务动作验证领域事件的发布与订阅处理链路。

### 9.2 查询与分析

1. 用户输入自然语言查询后，系统可生成正确的只读查询与结果展示。
2. 用户输入分析型问题后，系统可返回结果、理由和相应图表或列表。

### 9.3 业务操作

1. 用户发起录入或状态变更时，系统通过受控技能执行写操作。
2. 状态变更前会进行规则与状态机校验。
3. 状态变更后会同步更新对话反馈与界面状态。
4. 若验证域配置了领域事件与订阅，终端用户在同一业务路径上触发的受控写或状态变更，应能体现与 **9.1 第 3 项**一致的发布与订阅行为（可与该演示场景复用，侧重用户视角）。

### 9.4 自愈与降级

1. 对于可修复的查询或工具参数错误，系统可在 2 次内自动修正并重试。
2. 对于不可修复错误，系统会明确反馈而不是静默失败。
3. AI 不可用时，系统仍可提供基础结构化操作能力。

## 10. 首个验证域

### 10.1 验证域选择

首个验证域采用合同管理，用于验证：

1. 元模型表达能力（含事件模型）。
2. 状态机与规则引擎联动能力。
3. 领域事件与订阅在典型业务路径上的可用性。
4. AI 查询、分析和操作能力。
5. 混合交互体验。

### 10.2 典型实体

建议至少覆盖以下实体：

1. 客户
2. 员工
3. 产品
4. 合同
5. 发票

## 11. 实施建议

1. 先以单领域、单体部署、低并发目标完成 MVP。
2. 优先打通“元模型定义 -> 生成 -> 发布 -> AI 理解 -> 用户操作”的闭环。
3. 在 MVP 通过后，再扩展复杂流程、更多领域和更高性能架构。
