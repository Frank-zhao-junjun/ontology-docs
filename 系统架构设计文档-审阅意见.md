## 3. 建议澄清或补强（按优先级）

### 3.1 需求「三层」与 HLD「四层」的映射

**现象**：《最终需求文档》**3.1** 为：元模型层、AI 引擎层、混合交互层。HLD **3.1** 为：交互层、编排层、领域运行层、元模型与生成层。

**判断**：属于合理细化（编排与领域运行拆开、元模型与生成合并），但评审易问「为何与需求三层不一致」。

**建议**：在 HLD **§3.1** 末尾增加简短对照（表或一段话），说明例如：混合交互层 ↔ 交互层；AI 引擎层 ↔ 编排层及工具执行路径；元模型层 ↔ 元模型与生成层 + 领域运行层对模型语义的消费等，避免被误解为偏离需求。

### 3.2 「流程模型」在运行时的落点

**现象**：需求 **4.5** 明确流程模型用于 AI 编排策略（入口、上下文收集、校验、决策点）。HLD **§4.3** 已列编排器子模块，但未写明 **已发布流程模型** 如何驱动步骤与配置。

**建议**：在 **§4.3** 增加 1～2 句：编排器按已发布流程模型（及规则）组织步骤；或说明流程模型快照与意图分析 / 工具路由的配置关系，避免实现阶段仅依赖代码写死流程。

### 3.3 规则热加载 vs 模型发布

**现象**：需求 **4.4.2** 要求规则变更后热加载或低中断生效。HLD **§4.4 规则引擎**侧重「依据已发布规则版本执行」，未区分「随模型发布的版本」与「规则热更新」。

**建议**：补充一句：规则引擎在已发布快照基础上，支持需求所述的热加载或等价低中断刷新策略，与 LLD 实现策略一致。

### 3.4 命名与 Markdown 结构

- **§3.2** 为「API 网关层」，**§4.2** 为「API 接入层」。建议统一称谓，或在 **§4.2** 首句注明「即 §3.2 中的 API 网关层」。
- **标题层级**：当前 **§3 与 §3.1、§4 与 §4.1** 等均为 `##`，小节未缩进。建议将 **§X.Y** 改为 `###`，其下「职责」「边界」再降为 `####`，便于目录与长期维护。

### 3.5 文首「对应文档」

HLD 写「实施计划 v1.0」。若仓库中实施计划文件名/版本未标 v1.0，建议统一版本号，或改为「实施计划（见 `实施计划.md`）」以免引用不一致。
