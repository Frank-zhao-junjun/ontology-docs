# 本体驱动 AI 原生应用框架系统架构设计文档（HLD）最终稿

版本：v2.0-final  
日期：2026-03-28  
对应文档：最终需求文档 v2.2；实施计划（见 实施计划.md）

## 1. 文档目标与范围

本文档用于定义系统高层架构，作为后续详细设计与实现的上游约束，覆盖：

1. 分层架构与模块边界。
2. 生成期与运行期核心链路。
3. 数据架构、接口契约、安全与运维约束。
4. 可测试性、可观测性、演进路径与风险控制。

本文档聚焦架构，不展开代码级实现细节。

## 2. 架构原则

1. 模型先行：五类元模型是业务语义单一事实来源。
2. 受控执行：写操作必须经过领域技能，规则与权限不可绕过。
3. 生成与运行解耦：生成期产出骨架与注册，运行期执行编排与受控动作。
4. 可追溯：关键链路必须具备版本追踪、日志追踪与上下文追踪。
5. 渐进增强：先满足单域 MVP，再扩展跨域、多租户与跨进程事件。

## 3. 分层架构

### 3.1 逻辑四层

1. 交互层：三栏式混合交互前端与动作协议消费。
2. 编排层：AI 编排器、上下文管理、工具路由、自愈执行。
3. 领域运行层：领域技能、规则引擎、状态机、事件运行时。
4. 元模型与生成层：五类元模型管理、发布流程、生成器。

### 3.2 与需求三层映射说明

为与需求文档三层表述保持一致，映射关系如下：

| 需求三层 | 本文四层映射 |
| --- | --- |
| 混合交互层 | 交互层 |
| AI 引擎层 | 编排层 + 领域运行层中的工具执行与受控运行能力 |
| 元模型层 | 元模型与生成层 + 领域运行层对已发布模型语义的消费 |

结论：四层是三层在实现视角下的细化，不构成偏离。

### 3.3 高层组件

1. 用户端
2. 交互前端
3. API 网关层
4. AI 编排器
5. 工具执行器
6. 领域技能服务
7. 规则引擎
8. 状态机引擎
9. 事件发布订阅运行时（进程内）
10. 元模型管理与发布服务
11. 生成器服务
12. 关系型数据库（元数据、业务数据、会话与日志）

## 4. 核心模块设计

### 4.1 交互层

#### 职责

1. 承载对话与结构化界面联动。
2. 消费动作协议并驱动页面状态。
3. 管理会话状态、聚焦实体与用户可见执行状态。

#### 边界

1. 不直接执行写入。
2. 不承担规则解释与业务决策。

### 4.2 API 网关层

说明：本节即原文 API 接入层，命名统一为 API 网关层。

#### 职责

1. 提供统一入口（对话、实体、模型、管理）。
2. 执行认证、授权、输入校验、限流与审计打点。
3. 路由到编排层或领域运行层。

#### 边界

1. 不承载复杂编排。
2. 不承载规则语义解释。

### 4.3 AI 编排层

#### 子模块

1. 上下文管理器：会话历史、聚焦实体、最小必要上下文装配。
2. 意图分析器：查询、分析、导航、操作意图识别。
3. 语义注入器：装配五类元模型已发布快照。
4. 流程策略解释器：依据已发布流程模型组织上下文收集、校验与决策步骤。
5. 工具路由器：按策略选择查询、技能、界面、图表等工具。
6. 响应整合器：统一输出消息与动作。
7. 自愈执行器：对可修复错误最多重试 2 次。

#### 流程模型运行时落点

1. 编排层必须消费已发布流程模型快照，而非代码硬编码流程。
2. 流程模型与规则模型共同约束工具路由与执行顺序。
3. 关键步骤的决策依据与执行结果进入日志追踪。

#### 边界

1. 不直接写库。
2. 不绕过领域技能、规则、状态机与权限。

### 4.4 领域运行层

#### 领域技能子系统

1. 承载全部受控写操作。
2. 执行参数校验与权限校验。
3. 联动规则引擎、状态机与事件发布。

#### 规则引擎子系统

1. 支持五种规则类型。
2. 默认基于已发布规则版本执行。
3. 支持规则热加载或等价低中断刷新策略。
4. 规则执行结果结构化返回并可观测。

#### 状态机子系统

1. 基于简化状态机执行流转。
2. 支持条件判断与权限检查。
3. 状态变化记录日志，并可触发领域事件。

#### 事件运行时子系统

1. MVP 采用进程内事件总线。
2. 支持同步与异步派发模式。
3. 支持一事件多订阅、一订阅多事件。
4. 订阅执行失败必须可记录、可观测，禁止静默丢弃。

### 4.5 元模型与生成层

#### 元模型管理

1. 管理五类元模型定义、版本与发布。
2. 提供发布前结构校验与一致性校验。
3. 向运行时提供已发布快照读取能力。

#### 生成器

1. 根据已发布模型生成数据结构、接口骨架、页面骨架。
2. 生成事件类型注册与订阅绑定占位。
3. 所有生成物包含来源版本标识。

## 5. DDD 与限界上下文

### 5.1 领域对象映射

1. 实体：合同、客户、发票等具备唯一标识与生命周期对象。
2. 值对象：金额、日期区间、地址等无独立标识对象。
3. 聚合根：如合同聚合，统一约束聚合内一致性。
4. 领域服务：跨实体协作逻辑由领域服务承载。
5. 领域事件：业务动作与状态变更后发布。

### 5.2 限界上下文建议

1. 元模型管理上下文。
2. 业务操作上下文。
3. 会话与编排上下文。
4. 观测与审计上下文。

上下文集成方式：优先 API 契约与进程内事件协作。

## 6. 关键流程设计

### 6.1 模型发布流程（生成期）

1. 提交模型变更。
2. 结构与一致性校验。
3. 生成产物与迁移计划。
4. 发布版本并登记快照。
5. 运行时刷新索引与缓存。

输出：可追溯的发布版本与生成产物。

### 6.2 对话查询流程（运行期只读）

1. 用户发起查询。
2. 编排器装配上下文并识别意图。
3. 流程策略解释器确定步骤并路由查询工具。
4. 工具执行器执行只读查询与安全过滤。
5. 返回消息、动作与可视化配置。

失败处理：触发自愈重试，超限后结构化错误返回。

### 6.3 受控写流程（运行期写入）

1. 用户发起录入或状态变更。
2. 编排器选择技能调用。
3. 技能执行参数校验、权限校验、规则校验。
4. 状态机校验并执行持久化。
5. 发布领域事件并返回结果。

约束：任何写入不得绕过技能路径。

### 6.4 事件订阅流程

1. 发布领域事件。
2. 订阅注册表匹配订阅方。
3. 同步或异步执行订阅处理。
4. 记录单订阅执行结果。
5. 汇总指标并触发告警。

## 7. 数据架构

### 7.1 数据域

1. 元模型域：模型定义、版本、发布记录。
2. 业务域：业务实体独立物理表。
3. 会话域：会话、消息、工具调用记录。
4. 运行日志域：规则、状态机、事件订阅执行日志。

### 7.2 数据库实现画像（当前项目）

以下为当前项目推荐实现画像，属于架构实现决策，不回写为需求强制条款：

1. 关系型数据库采用 MySQL 画像。
2. 业务实体独立物理表，关键关系使用外键或等价约束。
3. 高频查询字段建立索引。
4. 日志与消息类大表采用归档或分表策略。
5. 明确事务边界与一致性策略。

### 7.3 一致性与事务

1. 业务写与状态变更在同一事务内保持一致。
2. 事件发布与事务关系需在 LLD 固化（同事务发布或事务后补偿）。
3. 若订阅失败不回滚主事务，必须提供补偿与告警。

## 8. 接口与协议

### 8.1 对话接口契约

输入：会话标识、用户输入、上下文参数。  
输出：消息体、动作列表、上下文更新。

### 8.2 动作协议契约

1. 页面打开与导航。
2. 弹窗。
3. 数据刷新。
4. 聚焦实体。
5. 消息提示。
6. 图表渲染。

动作协议需包含版本字段，用于兼容演进。

### 8.3 工具与技能契约

1. 统一参数 Schema。
2. 统一结果结构（成功、失败、错误明细）。
3. 参数验证、权限检查、超时控制与错误标准化。

### 8.4 元模型管理契约

1. 创建与更新。
2. 校验。
3. 发布。
4. 已发布版本查询。

## 9. 安全架构

### 9.1 身份与授权

1. 登录后访问。
2. 基于角色或等价抽象授权。
3. API 路径与技能路径双重权限生效。

### 9.2 执行安全

1. 查询工具默认只读。
2. 禁止编排层直接执行写 SQL。
3. 工具参数白名单与结构校验。
4. 防注入、防越权、防重放等安全控制。

### 9.3 传输与应用安全

1. 传输层启用 HTTPS。
2. 采用令牌机制承载会话认证信息。
3. 防 SQL 注入、XSS、CSRF 等常见风险。
4. 定期安全审计与漏洞扫描。

### 9.4 审计与追踪

1. 记录用户操作与执行路径。
2. 记录工具调用输入输出摘要与错误。
3. 记录规则命中、状态流转、事件订阅结果。

## 10. 非功能方案

### 10.1 可用性

1. AI 不可用时降级为纯 UI 路径。
2. 服务健康检查与故障告警。
3. 备份与恢复可演练。

### 10.2 可测试性

1. 通过依赖注入提升可测试性。
2. 提供模拟对象支持集成测试。
3. 自动化测试纳入持续集成。

### 10.3 性能与扩展

1. MVP 不将性能数字作为验收性 KPI。
2. 通过缓存、异步化与索引优化提升体验。
3. 预留跨进程事件与多租户扩展点。

### 10.4 可维护性

1. 模型与生成物可追溯。
2. 配置中心化与环境隔离。
3. 日志字段标准化与链路追踪一致。

## 13. 风险与对策

| 风险 | 级别 | 对策 |
| --- | --- | --- |
| 模型定义与运行时不一致 | 高 | 发布前一致性校验、版本快照强绑定 |
| AI 工具调用不稳定 | 高 | 工具强约束、自愈、超时与回退 |
| 事件订阅失败不可见 | 中 | 标准化日志、失败告警、补偿机制 |
| 动作协议频繁演进 | 中 | 协议版本字段与兼容层 |
| MVP 范围膨胀 | 高 | P0 和 P1 门禁、变更评审 |

## 14. 设计决策记录

1. 采用五类元模型统一语义源。
2. 受控写统一经领域技能。
3. MVP 事件模型采用进程内总线。
4. 性能数字不作为 MVP 验收 KPI。
5. 四层架构作为三层需求的实现细化。
6. 规则执行采用发布快照 + 热加载并行策略。

## 15. 架构验收清单

1. 五类元模型可版本化发布。
2. 生成期与运行期共享同一发布快照。
3. 写操作全部走技能链路。
4. 规则与状态机在关键路径强制执行。
5. 流程模型在编排层真实生效，非代码硬编码。
6. 事件发布与订阅可观测、可追溯。
7. 角色与权限在 API 与技能路径均生效。
8. 对话与动作协议联动稳定且有版本字段。
9. AI 降级路径可用。
10. 审计日志字段齐全。
