# 本体驱动 AI 原生应用框架系统架构设计文档（HLD）

版本：v1.0  
日期：2026-03-28  
对应文档：最终需求文档 v2.2、实施计划 v1.0

## 1. 文档目标

本文档定义系统高层架构设计，覆盖：

1. 架构分层与模块边界。
2. 核心业务与技术链路。
3. 数据与状态流转路径。
4. 对外接口边界与契约原则。
5. 非功能方案与风险控制。

本文档不展开代码级细节，代码级设计在后续 LLD 中给出。

## 2. 架构原则

1. 模型先行：所有业务语义以五类元模型为单一事实来源。
2. 受控执行：写操作必须经过领域技能，规则与权限不可绕过。
3. 生成与运行解耦：生成期负责骨架与注册，运行期负责编排与执行。
4. 可追溯：关键链路必须有日志、版本和上下文追踪。
5. 渐进增强：先满足单域 MVP，再扩展高并发与跨系统能力。

## 3. 总体分层架构

## 3.1 逻辑分层

系统由四个逻辑层组成：

1. 交互层：三栏式混合交互前端与动作协议消费。
2. 编排层：AI 编排器、上下文管理、工具路由与自愈。
3. 领域运行层：领域技能、规则引擎、状态机、事件运行时。
4. 元模型与生成层：五类元模型管理、发布流程、生成器。

## 3.2 高层组件图（文本表示）

1. 用户端
2. 混合交互前端
3. API 网关层
4. AI 编排器
5. 工具执行器
6. 领域技能服务
7. 规则引擎
8. 状态机引擎
9. 事件发布订阅运行时（进程内）
10. 元模型管理与发布服务
11. 生成器服务
12. 关系型数据库（元数据、业务数据、会话与日志）

## 4. 核心模块设计

## 4.1 交互层

### 职责

1. 承载对话交互与结构化 UI。
2. 消费动作协议并驱动页面状态。
3. 管理会话、聚焦实体与用户可见执行状态。

### 边界

1. 不直接执行业务写入。
2. 不直接解释元模型规则，只展示与收集输入。

## 4.2 API 接入层

### 职责

1. 提供统一请求入口（对话、实体、模型发布、管理接口）。
2. 执行鉴权、基础校验、速率限制与审计打点。
3. 路由到编排层或领域运行层。

### 边界

1. 不承载复杂业务编排。
2. 不持有规则解释逻辑。

## 4.3 AI 编排层

### 子模块

1. 上下文管理器：会话历史、聚焦实体、最小必要上下文装配。
2. 意图分析器：查询、分析、导航、操作意图识别。
3. 语义注入器：加载五类元模型已发布版本并构造模型上下文。
4. 工具路由器：选择并调用执行工具。
5. 响应整合器：汇总结果并输出消息与动作。
6. 自愈执行器：在可修复错误场景下最多重试 2 次。

### 边界

1. 不直接写库。
2. 不绕过领域技能、规则与权限。

## 4.4 领域运行层

### 领域技能子系统

1. 承载所有受控写操作。
2. 执行前校验参数与权限。
3. 联动规则引擎与状态机。
4. 在业务动作完成后可发布领域事件。

### 规则引擎子系统

1. 支持五种规则类型。
2. 依据已发布规则版本执行。
3. 输出结构化校验结果。

### 状态机子系统

1. 基于简化状态机执行流转。
2. 校验流转合法性与前置约束。
3. 记录状态变化日志。

### 事件运行时子系统

1. 仅支持进程内领域事件发布与订阅触发。
2. 支持一事件多订阅、一订阅多事件。
3. 订阅执行失败必须可观测、可记录，不得静默丢弃。

## 4.5 元模型与生成层

### 元模型管理

1. 管理五类元模型定义与版本。
2. 提供发布前校验与发布流程。
3. 对运行期提供已发布快照读取能力。

### 生成器

1. 根据已发布模型生成数据结构、接口骨架、页面骨架。
2. 为事件模型生成类型注册与订阅绑定占位。
3. 生成结果带版本追踪标识。

## 5. 关键流程设计

## 5.1 模型发布流程（生成期）

1. 管理员提交模型变更。
2. 触发结构校验与一致性校验。
3. 校验通过后生成产物与迁移计划。
4. 发布模型版本并登记快照。
5. 通知运行时刷新可用版本索引。

输出：可追溯的已发布模型版本与对应生成产物。

## 5.2 对话查询流程（运行期只读）

1. 用户发起自然语言查询。
2. 编排器装配上下文并识别意图。
3. 生成只读查询工具调用。
4. 执行器做安全过滤并查询数据。
5. 返回消息、图表配置或页面动作。

失败处理：进入自愈重试，超限则结构化报错。

## 5.3 受控写操作流程（运行期写入）

1. 用户触发录入或状态变更。
2. 编排器选择技能调用。
3. 技能执行参数校验与权限校验。
4. 规则校验通过后进入状态机或持久化。
5. 成功后发布领域事件并返回结果。

约束：任何写入不得绕过技能路径。

## 5.4 事件订阅处理流程

1. 事件发布器发布领域事件。
2. 订阅注册表解析匹配订阅方。
3. 按实现策略同步或异步触发订阅处理。
4. 记录每个订阅执行结果。
5. 汇总成功与失败统计用于监控。

## 6. 数据架构设计

## 6.1 数据域划分

1. 元模型域：五类模型定义、版本、发布记录。
2. 业务域：合同等业务实体独立物理表。
3. 会话域：会话上下文、消息、工具调用记录。
4. 运行日志域：规则校验、状态流转、事件订阅执行日志。

## 6.2 存储原则

1. 元数据统一管理。
2. 业务实体使用独立物理表，不采用统一 JSON/EAV 作为主存方案。
3. 关键关系使用外键或等价约束。
4. 长文本或快照采用分表或归档策略防止主链路膨胀。

## 6.3 一致性原则

1. 写事务内至少保证业务写与状态变更的一致性。
2. 事件发布与事务一致性策略在实现中明确（同事务发布或事务后补偿）。
3. 订阅失败不得影响已提交事务时，必须有补偿与告警策略。

## 7. 接口与契约边界

## 7.1 对话接口

输入：会话标识、用户输入、上下文参数。  
输出：消息体、动作列表、上下文更新。

## 7.2 动作协议

动作类型至少包括：

1. 页面打开/导航。
2. 弹窗。
3. 数据刷新。
4. 聚焦实体。
5. 消息提示。
6. 图表渲染。

## 7.3 技能接口

1. 统一参数 Schema。
2. 统一返回结构（成功、失败、错误明细）。
3. 明确权限需求声明。

## 7.4 元模型管理接口

1. 模型创建与更新。
2. 模型校验。
3. 模型发布。
4. 已发布版本查询。

## 8. 安全架构

## 8.1 身份与授权

1. 登录后访问。
2. 基于角色或等价抽象的授权。
3. 关键写操作和敏感查询执行权限检查。

## 8.2 执行安全

1. 查询工具默认只读。
2. 禁止编排层直接执行写 SQL。
3. 工具参数做白名单与结构校验。
4. 高风险操作要求二次确认策略（可配置）。

## 8.3 审计与追踪

1. 记录用户操作与系统执行路径。
2. 记录工具调用输入输出摘要与错误。
3. 记录规则命中、状态流转、事件订阅结果。

## 9. 非功能架构方案

## 9.1 可用性

1. AI 不可用时降级为纯 UI 操作路径。
2. 关键服务健康检查与故障告警。
3. 数据备份与恢复流程可演练。

## 9.2 可维护性

1. 版本化模型与生成产物追溯。
2. 模块化边界清晰，便于替换模型供应商或存储实现。
3. 标准化日志字段与链路追踪标识。

## 9.3 可扩展性

1. 通过模型扩展新实体与新事件类型。
2. 通过技能扩展新业务能力。
3. 后续可平滑扩展到跨进程事件总线与多租户。

## 10. 部署与环境架构

## 10.1 MVP 部署形态

1. 单体部署。
2. 单数据库实例。
3. 进程内事件处理。

## 10.2 环境分层

1. 开发环境。
2. 测试环境。
3. 预发布环境。
4. 生产环境。

## 10.3 配置管理

1. 模型供应商与参数配置化。
2. 权限与工具白名单配置化。
3. 自愈重试上限与超时配置化。

## 11. 技术风险与架构对策

| 风险 | 级别 | 架构对策 |
| --- | --- | --- |
| 模型定义与运行时不一致 | 高 | 发布前一致性校验、版本快照强绑定 |
| AI 调用不稳定 | 高 | 工具强约束、自愈与回退路径 |
| 事件订阅失败不可见 | 中 | 订阅执行日志标准化、失败告警 |
| 动作协议频繁演进 | 中 | 协议版本化与兼容层 |
| MVP 范围膨胀 | 高 | P0/P1 门禁、变更评审机制 |

## 12. 设计决策记录（初版）

1. 决策：采用五类元模型统一语义源。  
   理由：支持从定义到运行的一致性与可追溯性。

2. 决策：业务写操作统一经领域技能。  
   理由：保证规则、权限、状态机可控执行。

3. 决策：事件模型 MVP 采用进程内发布订阅。  
   理由：降低复杂度，先验证事件语义闭环。

4. 决策：性能数字暂不作为 MVP 验收 KPI。  
   理由：优先验证业务闭环与架构正确性。

## 13. 架构验收清单

1. 五类元模型是否均可版本化发布。
2. 生成期与运行期是否共享同一已发布模型快照。
3. 写操作是否全部走技能链路。
4. 规则与状态机是否在关键路径强制执行。
5. 事件发布与订阅是否可观测、可追溯。
6. AI 降级路径是否可用。
7. 对话与动作协议联动是否稳定。
8. 关键日志与审计字段是否齐全。
